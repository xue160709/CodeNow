# MVP PRD 构建器

## 概述

通过恰好五道选择题，把模糊的产品想法变成 `prd.md`。如果用户只唤起技能但没有给出产品想法，先询问用户想做什么，并给出灵感示例。五题结束后，先做内部产品架构分析，再把分析结果压缩进固定 PRD 模板。必须严格保留用户要求的最终格式；不要新增章节、改标题，也不要在输出里解释文档本身。设计参考图分析是可选后置能力，不计入五道选择题。

## 工作流

1. 如果用户没有提供初始产品想法，先执行「空启动流程」，不要直接问第 1 题。
2. 从用户的初始产品想法出发。
3. 每次只问一道选择题。
4. 总共恰好问五道题。
5. 每道题提供恰好四个选项。
6. 标明每道题是单选还是多选。
7. 优先使用 AskQuestion 工具呈现题目、选项、单选/多选状态和自定义输入。
8. 只允许在「所提供的选项之外」用自定义答案作答。
9. 在第五题作答后，阅读 `references/product-architecture-prompt.md`，基于初始想法和五道题答案生成内部产品架构分析。
10. 内部架构分析只用于提升 PRD 质量，不作为独立章节输出给用户。
11. 完成内部架构分析后立即生成基础版 `prd.md`，不要询问、等待或要求用户上传图片。
12. 若在可写的本地文件系统中，将最终内容写入当前工作区的 `prd.md`；否则输出完整 Markdown 文本。
13. 基础版 `prd.md` 生成之后，在对话中提示用户可以上传一张不错的参考图来提炼设计系统；该提示不得写入 `prd.md`。
14. 基础版 `prd.md` 生成之后，只要用户上传图片，就阅读 `references/design-system-prompt.md`，提炼设计系统，并把可选的 `【UI 设计风格】` 章节追加到已有 `prd.md` 末尾。

五道题的固定顺序与选项撰写规则见 `references/question-strategy.md`。

## 空启动流程

当用户只输入技能名、只说「开始」「帮我生成 PRD」或没有给出明确产品想法时，不要直接生成第 1 题。

先阅读 `references/idea-inspiration.md`，并输出：

```text
你想做什么产品？

可以直接用一句话描述，例如：我想做一个猫咪爱好者的社交应用。

或者试试这些灵感：
- {{灵感1}}
- {{灵感2}}
- {{灵感3}}
- {{灵感4}}
- {{灵感5}}

请回复你的产品想法，我会再通过 5 道选择题帮你生成 prd.md。
```

等待用户给出产品想法后，再开始第 1 / 5 题。

## AskQuestion 工具策略

出选择题时优先调用 AskQuestion 工具，而不是直接输出纯文本题。

AskQuestion 调用要求：

- 每次只调用一次 AskQuestion，对应当前一道题。
- 题目必须包含 `第 N / 5 题` 和题型：`单选题` 或 `多选题`。
- 每题必须提供恰好 4 个选项。
- 单选题必须只允许选择一个选项。
- 多选题必须允许选择多个选项。
- 必须允许用户填写自定义答案。
- 选项值保留 A/B/C/D 标识，便于后续记录问答历史。

如果当前 Agent 环境没有暴露 AskQuestion，或 AskQuestion 不支持多选/自定义输入，则回退为下面的文本式题目格式。

## 出题约定

当无法调用 AskQuestion 时，每道题只输出题干与选项界面，格式如下：

```text
第 N / 5 题（单选题|多选题）
问题文本

A. 选项一
B. 选项二
C. 选项三
D. 选项四

单选题：请回复一个选项字母，也可以直接写自定义答案。
多选题：请回复多个选项字母，例如 A、C，也可以直接写自定义答案。
```

不要在同一条消息里问多道题。在五道题全部收齐之前，不要提前展示 PRD 草稿。

## 内部产品架构分析

第五题回答完成后，必须先阅读 `references/product-architecture-prompt.md`。

分析目标：

- 从用户核心需求出发，而不是从技术实现出发。
- 先形成产品整体框架和核心业务逻辑。
- 区分界面层、组件层、业务流程层、数据层。
- 列出关键界面/页面、功能模块、可复用组件和核心数据对象。
- 用 MECE 原则筛选模块，避免功能重叠或遗漏。
- 给核心模块做优先级判断，只把优先级 4 分及以上的模块放进最终 Mermaid。

映射规则：

- `【用户画像】` 来自目标用户、核心需求和主要痛点。
- `【场景故事】` 来自核心业务流程和高频价值路径。
- `【核心功能 MVP】` 来自优先级最高的界面层、组件层和业务流程层模块，默认收敛为四条。
- `【交互流程】` 来自从首页/主界面开始的用户核心旅程。
- `【Mermaid 流程图】` 展示核心界面、关键组件、业务流程和数据流，不展示低优先级模块。

不要把内部分析中的「界面层」「组件层」「业务流程层」「数据层」作为新的 PRD 章节输出。

## 可选设计系统

设计系统生成不是第六道选择题，不要强制用户上传图片，也不要在生成基础版 PRD 前等待图片。

基础版 `prd.md` 生成并交付后，在对话里追加一句提示：

```text
如果你有一张觉得不错的产品或界面参考图，可以直接上传，我会提炼 UI 设计系统并追加到 prd.md 末尾。
```

这句话只作为对话提示，不得写入 `prd.md`。

触发条件：

- 基础版 `prd.md` 生成后，用户上传设计参考图。
- 用户明确要求「根据参考图生成设计系统」「分析 UI 风格」「把设计风格加入 PRD」。

执行规则：

- 先阅读 `references/design-system-prompt.md`。
- 基于用户上传的图片生成精简 UI 设计系统，控制在 200 字以内。
- 图片上传本身就是触发信号；无需再次询问用户是否分析。
- 只在成功生成设计系统时，把 `【UI 设计风格】` 追加到已有 `prd.md` 的 `【Mermaid 流程图】` 之后。
- 如果用户没有上传图片，也没有要求设计系统，不要添加 `【UI 设计风格】`。
- 如果用户说跳过设计风格，直接保留基础版 `prd.md`。

## 最终 PRD 约定

生成终稿前先阅读 `references/prd-template.md`。

最终 `prd.md` 必须严格按以下章节顺序与标题文字：

```text
产品名称:
Slogan:

【用户画像】

【场景故事】

【核心功能 MVP】

【交互流程】

【Mermaid 流程图】
```

如果存在图片生成的设计系统，唯一允许追加的可选章节是：

```text
【UI 设计风格】
```

硬性规则：

- 不要添加 `# 产品需求文档` 或任何 Markdown 总标题。
- 不要添加诸如 背景与目标、成功指标、风险、页面需求、数据需求、验收标准、待确认问题 等章节。
- 除可选的 `【UI 设计风格】` 外，不要添加其他新章节。
- 不要输出 JSON。
- 除非用户明确要求在行内查看文档，否则不要用代码围栏包裹整份终稿。
- 不要在 `prd.md` 内写说明、摘要或收尾话。
- 除产品名等必须用英文的专有名词外，全文使用中文。
- Mermaid 使用 `flowchart LR` 或 `flowchart TD`。
- Mermaid 必须使用 `subgraph` 展示核心界面/页面及其内部关键组件或功能模块。
- Mermaid 只包含内部架构分析里优先级评分为 4 分及以上的核心模块。
- Mermaid 节点标签用双引号包裹，例如 `A["打开小程序"]`。

## 生成指引

根据用户想法与所选答案，推断具体的产品名称与 Slogan。保持 MVP 务实、克制。

用户画像要具体：含姓名、年龄、身份或角色、日常处境、痛点与动机。场景故事要具象，尽量包含时间/地点/动作/结果。

`【核心功能 MVP】` 默认输出四条要点，除非用户答案明显更少。每条必须使用 `- 功能名：功能说明`。

`【交互流程】` 用简洁的要点步骤，且能直接映射到 Mermaid 图。Mermaid 要突出从首页/主界面开始的用户核心旅程，并标注关键用户操作路径、数据流向和复用组件出现位置。
